AWS Identity and Access Management

Що таке AWS IAM?

  • Глобальний сервіс AWS (без регіону)
  • Безкоштовний
  • Керує доступом до ресурсів AWS
  • Підтримує деталізовані дозволи
  • Інтегрується з усіма сервісами AWS

Ключові компоненти

  • Users Користувачі
  • Groups Групи
  • Roles Ролі
  • Policies Політики

Модель спільної відповідальності

AWS відповідає за:
  • Безпеку інфраструктури IAM
  • Глобальну доступність
  • Шифрування облікових даних
Ви відповідаєте за:
  • Створення та управління обліковими записами
  • Призначення мінімальних привілеїв
  • Ротацію ключів і паролів
  • Налаштування MFA

Root Account vs IAM Users

⚠️ Root Account

  • Створюється разом з акаунтом AWS
  • Необмежений доступ до всіх ресурсів
  • Email + пароль для входу
  • Не рекомендується використовувати щодня!

Використовувати тільки для:
– Налаштування білінгу
– Закриття акаунту
– Відновлення доступу

👤 IAM Users

  • Окремі облікові записи для людей/сервісів
  • Обмежені за правами
  • Можуть мати консольний та/або програмний доступ
  • Підтримують MFA
  • Унікальний ARN для кожного

🔑 Типи облікових даних

  • Пароль (консоль AWS)
  • Access Key ID + Secret Access Key (API/CLI)
  • SSH-ключі (CodeCommit)
  • Сертифікати X.509

IAM Групи

Що таке IAM Group?

  • Колекція IAM-користувачів
  • Групи отримують політики, не людини
  • Користувач може бути в кількох групах
  • Групи не можуть містити інші групи
  • Спрощують управління правами

Приклад структури

  • Admins — повний доступ
  • Developers — EC2, S3, Lambda
  • Security — CloudTrail, GuardDuty, читання
  • Billing — доступ до білінгу

Переваги групової моделі

Замість призначення прав кожному користувачу окремо — призначте права групі. При переведенні співробітника просто змініть членство у групі.

IAM Ролі

Навіщо потрібні ролі?

  • Тимчасові облікові дані (замість постійних ключів)
  • Для сервісів AWS (EC2, Lambda тощо)
  • Для міжакаунтного доступу
  • Для федеративних ідентичностей (SSO)
  • Для зовнішніх IdP (Google, Active Directory)

Типи ролей

  • Service Role — для сервісів AWS
  • Cross-account Role — між акаунтами
  • Web Identity Role — Cognito, OIDC
  • SAML 2.0 Role — корпоративний SSO

Як працює AssumeRole

  1. Суб'єкт (user/service) запитує роль через sts:AssumeRole
  2. AWS STS перевіряє trust policy
  3. Повертаються тимчасові ключі (Access Key + Secret + Session Token)
  4. Ключі діють від 15 хв до 12 год
  5. Після закінчення — автоматичне скасування

💡 Ролі безпечніші за постійні Access Keys — немає ризику витоку довгоживучих облікових даних

IAM Політики

Типи політик

  • Identity-based — прикріплені до user/group/role
  • Resource-based — прикріплені до ресурсу (S3 bucket policy)
  • Permissions Boundary — обмеження максимальних прав
  • SCP — Service Control Policies (AWS Organizations)
  • Session Policy — тимчасові обмеження для сесії

Правило оцінки

Implicit Deny → Explicit Deny → Explicit Allow

Якщо немає явного дозволу — доступ заборонений

Структура JSON-політики

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource":
        "arn:aws:s3:::my-bucket/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}

Managed vs Inline Policies

AWS Managed Policies

  • Створені та підтримуються AWS
  • Готові до використання
  • Автоматично оновлюються при появі нових сервісів
  • Приклади: AdministratorAccess, ReadOnlyAccess, AmazonS3FullAccess

Customer Managed Policies

  • Створені вами
  • Повний контроль і гнучкість
  • Можна прикріпляти до кількох ідентичностей
  • Рекомендовано для production

Inline Policies

  • Вбудовані безпосередньо в user/group/role
  • Видаляються разом з ідентичністю
  • Не можна перевикористати
  • Не рекомендуються для масштабних рішень

💡 Порада

Надавайте лише необхідні права. Уникайте * в Action або Resource без крайньої потреби.

Багатофакторна аутентифікація (MFA)

Фактори аутентифікації

  • 🔑 Те, що знаєш — пароль, PIN
  • 📱 Те, що маєш — токен, телефон, смарт-карта
  • 👁 Те, чим є — біометрія (відбиток, обличчя)
  • 🌍 Де знаходишся — геолокація, IP-адреса

Типи MFA в AWS

  • TOTP Virtual MFA (Google Authenticator, Authy)
  • TOTP Hardware MFA (фізичний токен)
  • FIDO2 Security Key (YubiKey, passkey)

Обов'язковий MFA через умову в політиці

{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "BoolIfExists": {
      "aws:MultiFactorAuthPresent":
        "false"
    }
  }
}

Ця умова блокує будь-які дії, якщо MFA не пройдено. Рекомендується для адміністраторських акаунтів.

Принцип найменших привілеїв (PoLP)

Визначення

Кожен суб'єкт (людина, сервіс, процес) повинен мати лише мінімально необхідні права для виконання своїх функцій — і не більше.

Як застосовувати в AWS

  • Починайте з мінімальних прав, додавайте при необхідності
  • Використовуйте IAM Access Analyzer
  • Перевіряйте невикористані права (IAM Last Accessed)
  • Уникайте * в Action та Resource
  • Регулярно переглядайте та видаляйте зайві права

Приклад: Lambda-функція

❌ Погана практика: 
"Action": "*",
"Resource": "*"
✅ Гарна практика: 
"Action": [
  "dynamodb:GetItem",
  "dynamodb:PutItem"
],
"Resource":
  "arn:aws:dynamodb:eu-west-1:
  123456789012:table/Orders"

Федеративна ідентифікація та SSO

Принцип федерації

Довіра до зовнішнього Identity Provider (IdP): замість окремих IAM-користувачів — єдина корпоративна ідентичність.

Протоколи та стандарти

  • SAML 2.0 — для корпоративних IdP (AD, Okta)
  • OIDC — для вебзастосунків (Google, GitHub)
  • OAuth 2.0 — авторизація API
  • AWS SSO (IAM Identity Center) — централізований вхід для кількох акаунтів

Потік SAML 2.0 у AWS

  1. Користувач входить у корпоративний портал (Okta, AD FS)
  2. IdP аутентифікує та генерує SAML assertion
  3. Браузер перенаправляє на AWS SSO endpoint
  4. AWS перевіряє assertion, припускає IAM роль
  5. Видаються тимчасові credentials

💡 Перевага: не потрібно створювати IAM user для кожного співробітника. Звільнення — лише в корпоративному IdP.

IAM у телекомунікаційних системах

Специфіка телекому

  • Велика кількість мережевих елементів та сервісів
  • Сервіси працюють цілодобово (24/7)
  • Мікросервісна архітектура (5G Core)
  • Вимоги ETSI, 3GPP для управління ідентичністю
  • Роуминг — міжоператорна довіра

Сценарії використання

  • NOC/SOC — обмежений доступ лише до моніторингу
  • CI/CD — автоматичні ролі без постійних ключів
  • API GW — авторизація API-запитів абонентів
  • BSS/OSS — розмежування між системами білінгу та управління

5G та ідентифікація

SUPI / SUCI — постійний та прихований ідентифікатор абонента

NF (Network Function) Identity — кожна мережева функція 5G Core має власну ідентичність (NF Instance ID)

OAuth 2.0 у 5G SBA — авторизація між NF у Service-Based Architecture

3GPP TS 33.501 визначає вимоги безпеки для 5G, включаючи управління ідентифікацією та авторизацією мережевих функцій

Типові атаки та вразливості IAM

Найпоширеніші атаки

  • 🔑 Витік Access Keys
    Ключі в публічних репозиторіях GitHub, у коді
  • 🎭 Privilege Escalation
    Використання прав для отримання ширших прав (iam:PassRole, iam:CreatePolicyVersion)
  • 🌐 Cross-Account Attack
    Некоректно налаштована trust policy дозволяє доступ з чужого акаунту
  • 📧 Credential Stuffing / Phishing
    Злам пароля консолі AWS через перебір або фішинг

Методи захисту

  • ✅ MFA для всіх адміністраторів
  • ✅ Ніколи не зберігати ключі в коді
  • ✅ Використовувати ролі замість ключів там, де можливо
  • ✅ AWS Secrets Manager для ротації ключів
  • ✅ CloudTrail + GuardDuty для виявлення аномалій
  • ✅ IAM Access Analyzer для пошуку надмірних прав
  • ✅ SCPs в AWS Organizations для жорстких меж
Реальний інцидент: У 2019 р. Capital One витік даних 100 млн клієнтів — через надмірні права IAM-ролі на EC2-інстансі.

Ескалація привілеїв в AWS IAM

Шляхи ескалації

iam:CreatePolicyVersion

Перезаписати існуючу managed policy, додавши "Action": "*"

iam:PassRole + ec2:RunInstances

Запустити EC2 з більш привілейованою роллю

iam:AttachUserPolicy

Прикріпити собі AdministratorAccess policy

Як виявляти

  • CloudTrail: відстежувати дії iam:* та sts:*
  • AWS Config: правила для виявлення надмірних прав
  • GuardDuty: виявлення підозрілих IAM-дій
  • IAM Access Analyzer: аналіз зовнішнього доступу

Превентивні заходи

  • Permissions Boundary — жорстке обмеження максимальних прав
  • SCP — заборона небезпечних дій на рівні організації
  • Регулярний аудит з IAM Credential Report

Кращі практики AWS IAM

🔒 Облікові дані

  • Увімкнути MFA для root і всіх адмінів
  • Ротувати Access Keys кожні 90 днів
  • Видаляти невикористані ключі та користувачів
  • Ніколи не вбудовувати ключі в код або конфіги

🎭 Ролі та права

  • Використовувати ролі для EC2, Lambda та інших сервісів
  • Принцип найменших привілеїв для кожної ідентичності
  • Не використовувати root для повсякденної роботи
  • Розмежовувати dev/staging/production акаунти (AWS Organizations)

📊 Аудит та моніторинг

  • Увімкнути CloudTrail у всіх регіонах
  • Налаштувати сповіщення на root-активність
  • Використовувати IAM Access Analyzer
  • Регулярно переглядати Credential Report

🏢 Організаційний рівень

  • AWS Organizations + SCP для жорсткого управління
  • AWS IAM Identity Center для централізованого SSO
  • Permission Boundaries для делегованого адміністрування
  • Визначити процедури управління ідентичністю (Joiner-Mover-Leaver)

AWS IAM Identity Center (SSO)

Можливості

  • Єдина точка входу для кількох AWS-акаунтів
  • Інтеграція з AD, Okta, Azure AD
  • Permission Sets — набори прав для груп
  • Тимчасові облікові дані через STS
  • Портал самообслуговування для користувачів
  • Підтримка SCIM для автоматичного провізіонування

Рекомендовано для

  • Компаній з >5 AWS-акаунтів
  • Корпоративних середовищ з Active Directory
  • Команд із частою ротацією персоналу

Zero Trust та сучасний IAM

Принципи Zero Trust

«Never trust, always verify»

  • Перевіряти ідентичність при кожному запиті
  • Мінімальні привілеї за замовчуванням
  • Постійний моніторинг та валідація
  • Мікросегментація доступу

IAM як основа Zero Trust

  • Ідентичність — новий периметр безпеки
  • Контекстна авторизація (IP, час, пристрій)
  • Безперервна оцінка ризиків сесій

Умови в IAM Policies

{
  "Condition": {
    "StringEquals": {
      "aws:RequestedRegion":
        "eu-west-1"
    },
    "IpAddress": {
      "aws:SourceIp": [
        "10.0.0.0/8"
      ]
    },
    "DateGreaterThan": {
      "aws:CurrentTime":
        "2024-01-01T00:00:00Z"
    },
    "Bool": {
      "aws:SecureTransport": "true"
    }
  }
}
